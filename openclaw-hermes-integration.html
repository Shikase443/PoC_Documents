OpenClaw と Hermes Agent の連携モデル

OpenClawを進行係、Hermes Agent群を作業者として使い、問い合わせ対応を半自動化するための設計整理です。 この記事では、まず最終的に落ち着いたモデルを説明し、その後に当初案とあい路、最後に今後の拡張構想をまとめます。

最終モデル 用語 現在の構成 実装 背景思想 当初案 あい路 今後

1. 最終的に落ち着いたモデル

最終的には、OpenClawがワークフローを制御し、Hermes Agent群へ単一責務の作業を順番に依頼するモデルに落ち着きました。

ざっくり言うと、OpenClawは「受付と進行係」です。人間からの問い合わせを受け取り、質問文を共有フォルダへ置き、Hermesの各agentに順番に作業を依頼します。 Hermes側は「調査する人」「回答を書く人」「最後に確認する人」として使います。

人間 問い合わせや相談をOpenClawへ渡す。
OpenClaw 受付と進行係。API呼び出しと進行管理を行う。
Hermes 調査、回答下書き、検証を担当する。
NFS 質問文や回答ファイルを置く共有フォルダ。
回答 OpenClawが最終結果を人間へ返す。

重要なのは、Hermesに長い段取りをすべて任せないことです。 OpenClaw側のスキルがAPI呼び出し、作業完了の確認、成果物ファイルの確認を行います。 Hermesには「調査だけ」「回答下書きだけ」「検証だけ」という短い仕事を渡します。

この構成では、OpenClawが制御プレーン、Hermesが作業プレーン、NFS PVが共有作業台、OpenAI互換APIが実行トリガーです。

最終フロー

1. 問い合わせ受付
OpenClawが問い合わせ本文を受け取り、inbox/<task_id>/task.md に保存します。
2. 調査依頼
OpenClawスキルが調査用テンプレートを読み、保存先などを埋め込んで agent-1 の /v1/runs に投入します。
3. 調査結果確認
OpenClawが run_id をポーリングし、research/result.md と sources.json を読み戻します。
4. 回答作成依頼
OpenClawスキルが回答作成用テンプレートを埋め込み、agent-2 に回答下書きを依頼します。
5. 検証依頼
OpenClawスキルが検証用テンプレートを埋め込み、agent-0 に最終確認を依頼します。
6. 完了
done/<task_id>/final.md と status.json を保存し、OpenClawが人間へ回答します。

2. 用語のかんたんな説明

この資料で出てくる専門用語を、今回の構成に合わせて言い換えると次のようになります。

用語 ここでの意味
Agent AIに役割を持たせた作業者です。今回なら「調査担当」「回答作成担当」「検証担当」のように分けています。
API 別のプログラムに依頼を出すための窓口です。OpenClawがHermesへ「この作業をして」と頼む時に使います。
/v1/runs Hermesに作業を開始させる入口です。依頼すると run_id という受付番号が返ります。
run_id 作業の受付番号です。この番号で「終わったか」「失敗したか」をあとから確認します。
ポーリング 作業が終わるまで、一定間隔で状態を見に行くことです。
NFS PV KubernetesのPodとOpenClawの両方から見える共有フォルダです。質問文や回答ファイルを置く場所です。
NodePort Kubernetes内のサービスをLAN内のIPとポートで呼べるようにする公開方法です。
テンプレート AIに渡す定型の指示文です。毎回同じ書き方にすることで動作を安定させます。

3. 現在の構成

現在の構成では、OpenClaw側に hermes-support-flow スキルを作り、そこからHermesの各agentを呼び出します。 OpenClawスキルは、Hermesに渡すプロンプトをテンプレートから作り、APIを呼び、完了まで待ち、ファイルを確認します。

OpenClaw 人間との対話、問い合わせ受付、API呼び出し、ポーリング、成果物確認を担当します。
agent-1 調査担当です。必要な確認事項、コマンド例、原因候補、追加情報を整理します。
agent-2 回答下書き担当です。調査結果をもとに、問い合わせ者へ返す文章を作ります。
agent-0 検証担当です。回答が問い合わせに答えているか確認します。
NFS PV OpenClawとHermesが共有する作業フォルダです。長文や成果物はここで受け渡します。
OpenAI互換API Hermesへの実行トリガーです。将来、同じようなAPIを持つ別agentへ差し替える余地を残します。

共有フォルダの構成

host path:

Hermes pod mount:
/artifacts

NodePortなので、通常は 10.0.1.10 から 10.0.1.13 のどのNode IPでも同じポートでアクセスできます。 スキルのデフォルトは 10.0.1.10 です。

4. 実装

OpenClaw側のスキルは、Hermesへ渡す指示をテンプレートとして持っています。 HermesにNFS上のテンプレートを読ませるのではなく、OpenClawスクリプトがテンプレートを読み、{{task_id}} や保存先パスを埋め込んでAPI inputに入れます。 

/home/haomei/.openclaw/workspace/skills/hermes-support-flow/
├── SKILL.md
├── scripts/hermes_support_flow.py
└── templates/
    ├── research-prompt.md
    ├── writer-prompt.md
    ├── verify-prompt.md
    └── task.md

スクリプトが担当すること

  • NFS上に inbox/<task_id>/task.md を作る。
  • 調査、回答作成、検証のテンプレートを読み込む。
  • テンプレート内の {{task_id}}、{{task_path}}、{{research_path}} などを埋める。
  • Hermes agentの /v1/runs に投入する。
  • /v1/runs/{run_id} をポーリングする。
  • 成果物ファイルを読み戻して確認する。
  • 完了時に done/<task_id>/status.json を作る。

実行例

python3 /home/haomei/.openclaw/workspace/skills/hermes-support-flow/scripts/hermes_support_flow.py \
  --input /path/to/question.txt

直接テキストを渡す例

python3 /home/haomei/.openclaw/workspace/skills/hermes-support-flow/scripts/hermes_support_flow.py \
  --text "KubernetesのPodがCrashLoopBackOffになる原因を切り分けたい"

5. 背景にある思想

この設計の根底には、KubernetesやOSSに近い価値観があります。 つまり、特定の製品に依存しすぎず、共通のインターフェースと明確な責務分担によって、構成要素を入れ替えられるようにしておくという考え方です。

Kubernetesでは、コンテナランタイム、CNI、CSI、Ingress Controller、Operator、監視基盤などを、標準的なAPIや契約に沿って組み合わせます。 どれか1つの実装に閉じず、要件や時代に合わせて差し替えられることが、Kubernetesエコシステムの強さです。 今回のOpenClawとHermesの連携でも、それと同じ発想をAIエージェントの世界に持ち込みたい、という意図があります。

ベンダーロックインを避けるための契約

重要なのは、どの製品を使うかではなく、次の契約を守れるかです。

  • 問い合わせ本文をファイルとして受け取れること。
  • API経由で作業を開始できること。
  • 作業状態を確認できること。
  • 調査結果、回答下書き、最終回答を決められた形式で保存できること。
  • 人間が必要に応じてファイルを確認・編集できること。

OpenClawもHermesも置き換え可能にする

Hermes側は、OpenAI API互換の入口とファイル入出力契約を守れるagentなら将来置き換えられます。 同じように、OpenClaw側も、スキルを作成でき、外部APIを呼び出し、ファイルを読み書きできるAIエージェントであれば置き換え可能です。

これは特定製品同士を強く結合するための構成ではありません。 進行係、調査担当、回答作成担当、検証担当という役割を、共通のAPIと共有ファイル契約でつなぐ、AIエージェントのエコシステム的な設計です。

Hermes kanbanを中核にしなかった理由

Hermesにはkanbanによるタスク連携機能があります。 しかし今回は、あえてkanbanを中核には置きませんでした。 理由は、kanbanを前提にするとワークフロー全体がHermes固有の機能に強く結びつき、将来別のAIエージェントへ置き換えにくくなるためです。

便利な製品固有機能を使うこと自体は悪くありません。 ただし今回の狙いは、OpenClaw、Hermes、将来の別エージェントを、共通のAPIと共有ファイル契約で接続することでした。 そのため、タスク管理の中核はkanbanではなく、inbox、research、outbox、done というファイル上の状態遷移に寄せました。

6. 当初やろうとしていたモデル

当初のモデルは、テクニカルサポート体制を模した Hermes Agent同士の自律連携でした。 OpenClawは問い合わせ者として入力を渡し、Hermes側の agent-0 がディスパッチャーとして全体を動かす想定でした。

例えるなら、OpenClawが「お客さんからの問い合わせを持ってくる人」で、agent-0が「サポート窓口の班長」です。 班長であるagent-0が、調査担当のagent-1と回答作成担当のagent-2に仕事を振り分ける、という発想でした。

当初モデル

OpenClaw
  -> agent-0
      -> agent-1
      -> agent-2
      -> final

agent-0がディスパッチ、API呼び出し、ポーリング、検証まで担当する。

最終モデル

OpenClaw Skill
  -> agent-1
  -> agent-2
  -> agent-0
  -> final

OpenClaw側の固定スクリプトがAPI呼び出し、ポーリング、成果物確認を担当する。

当初モデルの狙い

  • OpenClawは人間との対話に集中する。
  • Hermesはタスク処理、調査、回答作成に集中する。
  • NFS上のファイル契約にすることで、将来別製品へ置き換えやすくする。
  • API inputは短くし、長文や成果物はNFSで受け渡す。
  • 製品固有の機能ではなく、APIとファイルという単純な契約を中心に据える。

7. あい路事項

実験により、当初モデルは概念としては自然でしたが、実運用ではいくつかのあい路がありました。

1. agent-0が計画で止まりやすい

agent-0は問い合わせ本文と手順書を読み、次に実行すべきコマンドを説明するところまでは安定していました。 しかし「このコマンドを実行します」と言ったまま、実際のterminal実行に進まないことがありました。 

実行するアクション:
/artifacts/bin/hermes-run agent-1 <task_id> research

このコマンドを実行し、結果を待ちます。

人間なら「では実行します」と言ったあとに手を動かしますが、LLMは文章で計画を書いた時点で満足してしまうことがあります。 今回はこの性質が、連携処理の途中停止につながりました。

2. ローカルファイルをWeb取得しようとする

/artifacts/templates/research-worker.md のようなローカルファイルを、Hermesが file:///artifacts/... に変換し、 web_extract で読もうとしてブロックされることがありました。 

Blocked: URL targets a private or internal network address
url: file:///artifacts/templates/research-worker.md

対策として、Hermes workerにNFS上のテンプレートを読ませるのではなく、OpenClawスキルがテンプレートを読み、API inputへ埋め込む方式に変更しました。

3. terminal実行環境にAPIキーが渡らない

Hermes本体プロセスには API_SERVER_KEY があり、Pod間API通信も成功していました。 しかし Hermes の terminal tool 実行環境では API_SERVER_KEY が見えず、agent-1 API呼び出しが401になることがありました。 

HTTP 401
{"error":{"message":"Invalid API key","code":"invalid_api_key"}}

対策として、Hermesの設定に terminal.env_passthrough を追加しました。 

terminal:
  env_passthrough:
  - API_SERVER_KEY

4. 自己申告と成果物がズレる

agentが「保存しました」と返しても、実際にはNFS上にファイルが存在しないことがありました。 AIの返答は「自己申告」であり、実際にファイルがあるかどうかとは別です。 そのため最終モデルでは、OpenClaw側のスクリプトがファイルの存在と中身を確認します。

5. 製品固有機能を中核にすると移植性が下がる

Hermes kanbanのような製品固有機能を使えば、単一製品内では自然に連携できる可能性があります。 しかし、それを中核にすると「Hermesだから動く」構成になり、OpenAI API互換の別エージェントや、別のスキル型エージェントへ置き換えにくくなります。

これらのあい路は、Hermesが悪いというより、LLMに長い実行制御を任せた時の揺れです。 繰り返しで改善する余地はありますが、基幹フローは固定スクリプトに寄せる方が安定します。

8. 今後の拡張構想

現在の調査用指示では、RAG用の社内ドキュメントや過去事例データベースはまだ組み込んでいません。 いまの段階では、問い合わせ本文、Web検索、明示されたファイル、調査担当へのプロンプトを中心に回答を作る構成です。

今後は、社内の障害事例、運用手順書、設計資料、FAQ、過去の問い合わせ回答などを検索対象に加えることで、 より実務に近いテクニカルサポート基盤にできます。

RAG用ドキュメントの活用

RAGは、AIが回答する前に関連文書を検索し、その内容を根拠として回答に使う仕組みです。 OpenClawスキルまたは調査担当agentの前段で、社内文書から関連情報を検索し、調査プロンプトへ「参考資料」として渡す形が考えられます。

問い合わせ 利用者からの質問文。
社内文書検索 過去事例、手順書、FAQを検索。
調査agent 検索結果とWeb情報を整理。
回答agent 根拠に基づく回答を作成。
検証agent 問い合わせへの回答になっているか確認。

分野別エージェントセット

もう一つの検証テーマは、4エージェント構成を分野別に持つことです。 Kubernetes、ネットワーク、業務アプリ、社内システム、セキュリティなどの領域ごとに、 専用の調査・回答・検証テンプレートを持つ構成が考えられます。

分野 想定する知識源 向いている問い合わせ
Kubernetes クラスタ運用手順、障害対応記録、kubectl例、公式ドキュメント Pod障害、Deployment、ConfigMap、Secret、リソース不足
ネットワーク 構成図、IPアドレス管理、FW/NAT設定、監視ログ 疎通不可、名前解決、ルーティング、ポート開放
業務アプリ アプリ仕様書、FAQ、リリースノート、過去問い合わせ 利用方法、エラー、仕様確認、設定変更
セキュリティ ポリシー、監査手順、インシデント対応記録 権限、監査、脆弱性対応、アクセス制御

AIエージェントのエコシステムとしての狙い

この構成の本質は、OpenClawとHermesという2製品の固定連携ではありません。 「進行係」「調査担当」「回答作成担当」「検証担当」という役割を、複数のAIエージェントが共通のAPIとファイル契約で受け渡すことにあります。

進行係はOpenClawでなくても、スキルやワークフローを作れるエージェントなら置き換えられます。 作業者もHermesでなくても、OpenAI API互換の入口を持ち、指定されたファイルを読み書きできるエージェントなら置き換えられます。 このように、AIエージェント同士を疎結合にして、用途に応じて組み合わせを変えられるエコシステムを目指しています。

これはKubernetesやOSSの世界で重視されてきた「一社の製品に閉じない」「標準的な接続面を残す」「実装を選べるようにする」という思想に近いものです。 特定の時点で最も使いやすい製品を選びつつ、将来もっと良い実装が出てきた時に、全体を作り直さずに差し替えられる余地を残します。 

進行係エージェント
  -> 調査エージェント
  -> 回答作成エージェント
  -> 検証エージェント
  -> 共有ファイル契約で成果物を受け渡す
まずは現在の1セット構成を安定させ、その後に「Kubernetes向け」など対象を絞った分野別セットを1つ追加して検証するのが現実的です。
