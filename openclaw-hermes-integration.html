OpenClaw と Hermes Agent 連携モデル整理

当初目指した「Hermes同士の自律連携」、そこで見えたあい路、最終的に落ち着いた「OpenClaw主導のワークフロー制御」を整理するための資料です。

目的 用語 当初モデル あい路事項 最終モデル 実装要素 運用方針 今後

1. 目的

OpenClawを人間との問い合わせ窓口にし、Hermes Agent群を調査・回答作成・検証に使うことで、問い合わせ対応を半自動化することが目的です。

ざっくり言うと、OpenClawを「受付と進行係」、Hermesを「調べる人・文章を書く人・確認する人」として分担させる構成です。 難しい仕組みに見えますが、やっていることは「質問を置く」「調べてもらう」「回答を書いてもらう」「確認する」の4段階です。

人間 問い合わせや相談をOpenClawへ渡す。
OpenClaw 受付と進行係。誰に何を頼むかを決める。
Hermes 調査、回答下書き、確認を担当する作業者。
NFS 質問文や回答ファイルを置く共有フォルダ。
回答 OpenClawが最終結果を人間へ返す。

最初は Hermes の agent-0 をディスパッチャーにして、agent-1 と agent-2 へ作業を割り振る構成を試しました。 しかし実験の結果、Hermesにワークフロー制御まで任せるより、OpenClaw側でAPI呼び出しとポーリングを固定実装した方が安定することが分かりました。

結論として、OpenClawが制御プレーン、Hermesが作業プレーン、NFS PVが共有作業台、OpenAI互換APIが実行トリガー、というモデルに落ち着きました。

2. 用語のかんたんな説明

この資料で出てくる専門用語を、今回の構成に合わせて言い換えると次のようになります。

用語 ここでの意味
Agent AIに役割を持たせた作業者です。今回なら「調査担当」「回答作成担当」「検証担当」のように分けています。
API 別のプログラムに依頼を出すための窓口です。OpenClawがHermesへ「この作業をして」と頼む時に使います。
/v1/runs Hermesに作業を開始させる入口です。依頼すると run_id という受付番号が返ります。
run_id 作業の受付番号です。この番号で「終わったか」「失敗したか」をあとから確認します。
ポーリング 作業が終わるまで、一定間隔で状態を見に行くことです。
NFS PV KubernetesのPodとOpenClawの両方から見える共有フォルダです。質問文や回答ファイルを置く場所です。
NodePort Kubernetes内のサービスをLAN内のIPとポートで呼べるようにする公開方法です。
テンプレート AIに渡す定型の指示文です。毎回同じ書き方にすることで動作を安定させます。

3. 当初やりたかった連携モデル

当初のモデルは、テクニカルサポート体制を模した Hermes Agent同士の自律連携でした。 OpenClawは問い合わせ者として入力を渡し、Hermes側の agent-0 がディスパッチャーとして全体を動かす想定でした。

例えるなら、OpenClawが「お客さんからの問い合わせを持ってくる人」で、agent-0が「サポート窓口の班長」です。 班長であるagent-0が、調査担当のagent-1と回答作成担当のagent-2に仕事を振り分ける、という発想でした。

OpenClaw 問い合わせ者。人間から受けた質問をNFSの inbox に置き、agent-0へ処理を依頼する。
agent-0 ディスパッチャー兼検証担当。問い合わせを読み、agent-1/2へAPIで委譲し、最終回答を確認する。
agent-1 検索・調査担当。必要な情報を調べ、調査結果をNFSへ保存する。
agent-2 回答作成担当。調査結果から回答下書きを作る。
NFS PV inbox、research、outbox、done を共有する作業領域。
Hermes API agent間の実行指示に使う /v1/runs API。

想定フロー

1. 問い合わせ投入
OpenClawが /artifacts/inbox/<task_id>/task.md に不定形の問い合わせ本文を置く。
2. ディスパッチ
OpenClawが agent-0 に「この inbox を処理して」と短く依頼する。
3. 調査
agent-0 が agent-1 の /v1/runs を呼び、調査結果を research/ に保存させる。
4. 回答作成
agent-0 が agent-2 の /v1/runs を呼び、回答下書きを outbox/ に保存させる。
5. 検証
agent-0 が回答下書きを検証し、問題なければ done/final.md に保存する。

当初モデルの狙い

  • OpenClawは人間との対話に集中する。
  • Hermesはタスク処理、調査、回答作成に集中する。
  • NFS上のファイル契約にすることで、将来別製品へ置き換えやすくする。
  • API inputは短くし、長文や成果物はNFSで受け渡す。

4. あい路事項

実験により、当初モデルは概念としては自然でしたが、実運用ではいくつかのあい路がありました。

期待していた動き

agent-0が手順書を読み、必要なコマンドを自分で実行し、agent-1とagent-2へ順番に作業を依頼する。

実際に起きたこと

agent-0は手順を理解して「次に実行します」と言うが、そこで止まったり、ファイルをWebとして読もうとしたりした。

1. agent-0が計画で止まりやすい

agent-0は問い合わせ本文と手順書を読み、次に実行すべきコマンドを説明するところまでは安定していました。 しかし「このコマンドを実行します」と言ったまま、実際のterminal実行に進まないことがありました。

人間なら「では実行します」と言ったあとに手を動かしますが、LLMは文章で計画を書いた時点で満足してしまうことがあります。 今回はこの性質が、連携処理の途中停止につながりました。 

実行するアクション:
/artifacts/bin/hermes-run agent-1 <task_id> research

このコマンドを実行し、結果を待ちます。

これは、ワークフロー制御をLLMの自律実行に任せると、計画と実行の境界で止まる可能性があることを示しています。

2. ローカルファイルをWeb取得しようとする

/artifacts/templates/research-worker.md のようなローカルファイルを、Hermesが file:///artifacts/... に変換し、 web_extract で読もうとしてブロックされることがありました。

これは「同じファイルを読む」でも、ローカルファイルとして読むのか、Webページとして取りに行くのかをAIが取り違えた例です。 /artifacts は共有フォルダなので、Webアクセスではなく通常のファイル読み取りで扱う必要があります。 

Blocked: URL targets a private or internal network address
url: file:///artifacts/templates/research-worker.md

対策として、Hermes workerにNFS上のテンプレートを読ませるのではなく、OpenClawスキルがテンプレートを読み、API inputへ埋め込む方式に変更しました。

3. terminal実行環境にAPIキーが渡らない

Hermes本体プロセスには API_SERVER_KEY があり、Pod間API通信も成功していました。 しかし Hermes の terminal tool 実行環境では API_SERVER_KEY が見えず、agent-1 API呼び出しが401になることがありました。

これは「建物の受付には鍵があるが、作業部屋には鍵が渡っていない」ような状態です。 Hermes本体は鍵を持っていても、コマンドを実行する別環境に鍵が渡っていなければ、API呼び出しは認証に失敗します。 

HTTP 401
{"error":{"message":"Invalid API key","code":"invalid_api_key"}}

対策として、Hermesの設定に terminal.env_passthrough を追加しました。 

terminal:
  env_passthrough:
  - API_SERVER_KEY

4. 自己申告と成果物がズレる

agentが「保存しました」と返しても、実際にはNFS上にファイルが存在しないことがありました。 このため、成果物は必ずOpenClaw側で読み戻して確認する必要があります。

ここは重要です。AIの返答は「自己申告」であり、実際にファイルがあるかどうかとは別です。 そのため最終モデルでは、OpenClaw側のスクリプトがファイルの存在と中身を確認します。

これらのあい路は、Hermesが悪いというより、LLMに長い実行制御を任せた時の揺れです。 繰り返しで改善する余地はありますが、基幹フローは固定スクリプトに寄せる方が安定します。

5. 最終的に落ち着いた連携モデル

最終モデルでは、OpenClawに hermes-support-flow スキルを作成し、OpenClawがワークフロー制御を担当します。 Hermes agentは、それぞれ単一責務のworkerとして扱います。

つまり、OpenClawが「進行表を持っている係」になりました。 Hermesには長い段取りを考えさせず、「調査だけ」「回答下書きだけ」「検証だけ」という短い仕事を順番に依頼します。

当初モデル

OpenClaw
  -> agent-0
      -> agent-1
      -> agent-2
      -> final

agent-0がディスパッチ、API呼び出し、ポーリング、検証まで担当する。

最終モデル

OpenClaw Skill
  -> agent-1
  -> agent-2
  -> agent-0
  -> final

OpenClaw側の固定スクリプトがAPI呼び出し、ポーリング、成果物確認を担当する。

最終フロー

1. 受付 OpenClawが質問文を共有フォルダへ保存。
2. 調査 agent-1へ調査だけを依頼。
3. 下書き agent-2へ回答作成だけを依頼。
4. 検証 agent-0へ回答確認だけを依頼。
5. 返答 OpenClawが最終回答を人間へ返す。
1. OpenClaw
問い合わせ本文を受け取り、NFSの inbox/<task_id>/task.md に保存する。
2. OpenClaw Skill
research-prompt.md を読み、{{task_id}} や保存先を埋め込んで agent-1 へ /v1/runs 投入する。
3. agent-1
調査結果を research/<task_id>/result.md、参照情報を sources.json に保存する。
4. OpenClaw Skill
agent-1の run_id をポーリングし、成果物を読み戻して確認する。
5. agent-2
埋め込み済みの writer-prompt.md を受け取り、outbox/<task_id>/draft.md に回答下書きを保存する。
6. agent-0
埋め込み済みの verify-prompt.md を受け取り、回答下書きを検証し、done/<task_id>/final.md を作る。
7. OpenClaw Skill
done/status.json にrun_idと成果物パスを書き、OpenClawが人間へ結果を返す。

6. 実装要素

OpenClawスキル

スキルは /home/haomei/.openclaw/workspace/skills/hermes-support-flow に配置しています。 SKILL.md はOpenClaw向けの薄い説明にし、Hermesへ渡す実プロンプトは templates/ に分離しました。 

/home/haomei/.openclaw/workspace/skills/hermes-support-flow/
├── SKILL.md
├── scripts/hermes_support_flow.py
└── templates/
    ├── research-prompt.md
    ├── writer-prompt.md
    ├── verify-prompt.md
    └── task.md

スクリプトの役割

スクリプトは、この連携の「確実に実行する部分」です。 AIに任せると揺れやすいAPI呼び出し、待ち合わせ、ファイル確認を、普通のプログラムとして決め打ちで行います。

  • NFS上に inbox/<task_id>/task.md を作る。
  • テンプレート内の {{task_id}}、{{task_path}}、{{research_path}} などを埋める。
  • Hermes agentの /v1/runs に投入する。
  • /v1/runs/{run_id} をポーリングする。
  • 成果物が存在しない場合はAPI outputをフォールバック保存する。
  • 完了時に done/<task_id>/status.json を作る。

Hermes API

Hermes APIは、Hermesに作業を頼むための入口です。 agent-1、agent-2、agent-0 それぞれに別の入口を用意し、OpenClawスキルが順番に呼び出します。

agent NodePort 役割
agent-0 10.0.1.10:30242 最終検証
agent-1 10.0.1.10:30243 調査
agent-2 10.0.1.10:30244 回答下書き

NodePortなので、通常は 10.0.1.10 から 10.0.1.13 のどのNode IPでも同じポートでアクセスできます。 スクリプトのデフォルトは 10.0.1.10 です。

NFS PV

NFS PVは共有フォルダです。OpenClaw側ではホストの通常ディレクトリとして見え、Hermes側では /artifacts として見えます。 これにより、長文の問い合わせや回答ファイルをAPI本文に詰め込まずに受け渡しできます。 

host path:
/home/haomei/k8s-nfs/openclaw-hermes

Hermes mount:
/artifacts

7. 運用方針

基本方針

  • Hermesに複数ステップの実行制御を任せない。
  • OpenClaw側のスキルでAPI呼び出し、ポーリング、成果物確認を固定化する。
  • Hermesへ渡すプロンプトはスキル内テンプレートから埋め込む。
  • NFS上のファイルを成果物の正とする。
  • agentの「保存した」という自己申告は信用しすぎず、必ずファイルを確認する。

問い合わせ処理の実行例

python3 /home/haomei/.openclaw/workspace/skills/hermes-support-flow/scripts/hermes_support_flow.py \
  --input /path/to/question.txt

APIキー

OpenClawスクリプトは HERMES_API_KEY、API_SERVER_KEY、~/.openclaw/hermes-api-key、Kubernetes Secret の順にAPIキーを探します。 キー値はログや回答に出さない運用にします。

今後の改善候補

  • Hermes APIキーを専用の低権限トークンに分ける。
  • OpenClawスキルのログ保存先を決める。
  • 失敗時の再試行回数とタイムアウトを設定化する。
  • 問い合わせ種別ごとにテンプレートを分ける。
  • 最終回答の品質チェックをルール化する。

8. 今後の拡張構想

現在の調査用指示では、RAG用の社内ドキュメントや過去事例データベースはまだ組み込んでいません。 いまの段階では、問い合わせ本文、Web検索、明示されたファイル、調査担当へのプロンプトを中心に回答を作る構成です。

今後は、社内の障害事例、運用手順書、設計資料、FAQ、過去の問い合わせ回答などを検索対象に加えることで、 より実務に近いテクニカルサポート基盤にできます。

RAG用ドキュメントの活用

RAGは、AIが回答する前に関連文書を検索し、その内容を根拠として回答に使う仕組みです。 今回の構成に入れる場合、OpenClawスキルまたは調査担当agentの前段で、社内文書から関連情報を検索し、 調査プロンプトへ「参考資料」として渡す形が考えられます。

問い合わせ 利用者からの質問文。
社内文書検索 過去事例、手順書、FAQを検索。
調査agent 検索結果とWeb情報を整理。
回答agent 根拠に基づく回答を作成。
検証agent 問い合わせへの回答になっているか確認。

社内文書を使う場合は、回答に「どの文書を根拠にしたか」を残すことが重要です。 参照元が曖昧だと、もっともらしいが根拠の薄い回答になりやすいためです。

分野別エージェントセット

もう一つの検証テーマは、4エージェント構成を分野別に持つことです。 たとえば Kubernetes、ネットワーク、業務アプリ、社内システム、セキュリティなどの領域ごとに、 専用の調査・回答・検証テンプレートを持つ構成が考えられます。

分野 想定する知識源 向いている問い合わせ
Kubernetes クラスタ運用手順、障害対応記録、kubectl例、公式ドキュメント Pod障害、Deployment、ConfigMap、Secret、リソース不足
ネットワーク 構成図、IPアドレス管理、FW/NAT設定、監視ログ 疎通不可、名前解決、ルーティング、ポート開放
業務アプリ アプリ仕様書、FAQ、リリースノート、過去問い合わせ 利用方法、エラー、仕様確認、設定変更
セキュリティ ポリシー、監査手順、インシデント対応記録 権限、監査、脆弱性対応、アクセス制御

複数セット連携のイメージ

分野別に4エージェントセットを持つ場合、OpenClawスキルが最初に問い合わせ内容を分類し、 適切な分野セットへ処理を渡します。 1つの問い合わせが複数分野にまたがる場合は、複数セットへ調査を依頼し、最後に統合回答を作ることも検証できます。 

OpenClaw
  -> 問い合わせ分類
      -> Kubernetes向けセット
      -> ネットワーク向けセット
      -> 業務アプリ向けセット
  -> 調査結果を統合
  -> 最終回答

この方式を取ると、1つの巨大な汎用agentにすべてを任せるより、分野ごとの知識、テンプレート、検証観点を分けられます。 将来的に製品を入れ替える場合も、分野別の入出力契約を維持しやすくなります。

まずは現在の1セット構成を安定させ、その後に「Kubernetes向け」など対象を絞った分野別セットを1つ追加して検証するのが現実的です。
